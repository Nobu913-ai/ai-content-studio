# TTS品質管理ガイド

ElevenLabs 日本語ナレーションの品質を維持・改善するための運用ガイド。

## 整形パイプライン

台本 → ナレーションテキストの変換は3段階で処理される:

1. **マーカー除去** — `[SECTION:]`, `[B-ROLL:]`, `**太字**` 等を除去
2. **TTS向け短文化** — 1文35字以内に分割、コロン→改行、`%`→パーセント
3. **読み辞書適用** — `config/pronunciation-dictionary.json` の alias を置換

## 読み辞書の運用

### ファイル

`config/pronunciation-dictionary.json`

### ルール

- **実際に誤読を確認した語のみ登録する**（予防的な全文かな化は禁止）
- 一般的な漢字（投資、動画、設定、商品 等）はTTSが正しく読めるので登録しない
- ひらがなの羅列はアクセント推定を崩すため、変換は最小限に
- Voice / Model を変更したら辞書の有効性を再検証する

### 誤読を見つけたときの手順

1. 音声ファイルを聴いて誤読箇所を特定
2. `config/pronunciation-dictionary.json` の `entries` に追加:
   ```json
   {
     "entries": {
       "元の表記": "正しい読み"
     }
   }
   ```
3. `acs narration <channel> <script>` で再生成して確認
4. 改善を確認したらコミット

### 登録例

```json
{
  "新NISA": "しんニーサ",
  "NISA": "ニーサ",
  "投資信託": "とうししんたく",
  "非課税": "ひかぜい",
  "元本割れ": "がんぽんわれ",
  "売却": "ばいきゃく"
}
```

## Voice / Model 比較テスト

### Voice 比較

```bash
acs tts-bench-voice <channel> <narration-txt> --voices <id1>,<id2>,<id3>
```

同一テキスト・同一モデルで複数 Voice を生成し、比較レポートを出力。

### Model 比較

```bash
acs tts-bench-model <channel> <narration-txt> --models <model1>,<model2>
```

同一テキスト・同一 Voice で複数モデルを生成し、比較レポートを出力。

### 比較の観点

| 観点 | 説明 |
|------|------|
| 誤読率 | 金融用語・制度語の読み間違い数 |
| イントネーション | 文全体の抑揚の自然さ |
| 文末の自然さ | 語尾が不自然に途切れていないか |
| 数字・制度語の安定性 | 「360万円」「非課税」等の読み |
| Shorts向けの抜け感 | カジュアルなトーンが出ているか |

### QAチェックポイント

毎回、最低限以下の3箇所は人が聴いて確認する:

1. **冒頭20秒** — 第一印象・Voice の雰囲気
2. **制度説明の中盤** — 金融用語の誤読・イントネーション
3. **最後のCTA前** — 文末の自然さ・全体の一貫性

## TTS向け台本のコツ

- 1文は20〜35文字以内
- 数字や制度名を含む文は特に短く
- コロン（：）は使わない（改行に置き換える）
- カギ括弧の連続は簡略化
- 強調箇所の前後に空行を入れる
